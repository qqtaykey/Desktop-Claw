# 1. 产品定义与定位

## 1.1 一句话定位

Desktop-Claw 是一个**桌面常驻的 AI Companion**——以悬浮球为最小入口，兼具陪伴感与轻实用能力，可逐步演化为有个性与成长感的桌面伙伴。

## 1.2 产品是什么

- 桌面常驻 AI Companion，不是需要专门打开的应用
- 悬浮球优先的轻量交互入口，不是全功能面板
- 先有用、再有个性，先成立、再成长

## 1.3 产品不是什么

- 不是全知全能的超级 Agent（不做高风险系统控制）
- 不是只能聊天的对话框（有文件处理和轻记录能力）
- 不是只有视觉皮肤的传统桌宠（有 AI-Native 的理解与响应）
- 不是重型自动化平台（不追求复杂工作流编排）

## 1.4 核心取舍

| 我们选择 | 而不是 |
|---------|--------|
| 做少一点，使用路径更自然 | 功能堆叠，入口混乱 |
| 做轻一点，用户更愿意高频用 | 大而全，每次交互成本高 |
| 先保证陪伴感 | 先追求任务完成率 |
| 为未来成长预留空间 | 一步到位做复杂系统 |

# 2. 目标用户

### 用户 A：AI 工具早期采用者
愿意尝试新交互形式，能接受桌面常驻工具，喜欢比网页聊天更短路径的交互。
→ 感知到的核心价值：**桌面入口**

### 用户 B：对陪伴型产品敏感的人
对角色感、存在感更敏感，喜欢常驻可互动的产品，不满足于纯效率工具。
→ 感知到的核心价值：**陪伴感**

### 用户 C：学习 / 工作陪跑型用户
需要轻量日常陪跑工具（记录、提醒、复盘），更在意"能不能随手用"和"有没有连续感"。
→ 感知到的核心价值：**轻陪跑**

# 3. 核心场景与用户旅程

## 3.1 场景一：桌面快速问答

**触发**：用户工作中遇到问题，不想切换应用
**旅程**：

1. 双击桌面悬浮球 → 水平条形输入框展开
2. 输入"JWT token 过期了怎么刷新" → 按 Enter
3. AI 回复以气泡形式出现在球上方（2-3 句简短回答）
4. 用户看完 → 双击球收起输入框，继续工作

**结束态**：问题解决，交互全程 < 10 秒，不离开当前工作上下文

## 3.2 场景二：文件处理

**触发**：用户收到一份长文档，想快速了解重点
**旅程**：

1. 用户拖拽 .md 文件到悬浮球上 → 球闪烁确认收到
2. Claw 显示气泡："收到了，帮你看看 📄"
3. 数秒后气泡显示摘要（3-5 句）和下一步建议
4. 用户如需深入 → 右键打开面板查看完整分析

**结束态**：用户 30 秒内拿到文件摘要，决定是否深入

## 3.3 场景三：轻记录与陪跑

**触发**：用户想记录今天要做的事
**旅程**：

1. 双击球 → "帮我记一下今天要做三件事：写周报、review PR、回邮件"
2. Claw 回复气泡："记好了 ✅ 今天三件事：1.写周报 2.review PR 3.回邮件"
3. 下午用户单击球 → Claw 冒出："还有两件没做完，加油～"
4. 晚上用户说"今天做了哪些" → Claw 通过面板回顾今日记录

**结束态**：用户的一天在 Claw 上有完整轨迹，形成连续感

## 3.4 场景四：陪伴轻触

**触发**：用户无明确需求，只是下意识点了一下球
**旅程**：

1. 单击悬浮球 → 球上方冒出一句短话（"在呢～""今天辛苦了！"）
2. 气泡 3 秒后自动消失
3. 用户没有后续操作

**结束态**：用户感受到"它在"，但没有被打扰

# 4. 交互层级设计

Desktop-Claw 的悬浮球入口提供三层交互，由浅到深：

### 1）单击 — 轻互动（陪伴层）
单击悬浮球，Claw 冒出一句短话（如"在呢～""今天怎么样？"）。话语以气泡形式出现在球上方，数秒后自动消失。MVP 阶段使用随机话术，后续接入 LLM 根据当日记忆生成短句。
定位：**零成本陪伴触达，让用户感受到"它活着"**。

### 2）双击 — 快速输入（效率层）
双击悬浮球，弹出水平延伸的条形输入框。再次双击收起。条形框自适应方向：球在屏幕右侧时向左展开，反之向右。AI 回复以气泡形式显示在球上方，最多 3 个，渐变透明（最旧最虚、最新最实），超出后最旧渐隐消失。
定位：**类 Spotlight 轻量交互，无需打开面板即可一问一答**。

### 3）右键 — 完整面板（深度层）
右键点击悬浮球，弹出上下文菜单：
- **打开面板**：完整聊天面板（多轮对话 + 历史回顾 + 记忆浏览）
- **设置**：API Key、模型配置等
- **退出**：关闭应用

面板与条形输入框**共享同一份对话历史**。
定位：**深度交互场景（长对话、回顾、配置）**。

### 交互层级总览

| 操作 | 触发 | 体验定位 | 回复方式 |
|------|------|----------|----------|
| 单击 | 轻互动气泡 | 陪伴感 | 球上方短句气泡，自动消失 |
| 双击 | 条形输入框 | 快速效率 | 球上方渐变气泡（最多 3 个） |
| 右键 | 上下文菜单 | 深度交互 | 完整面板内消息列表 |

### 对话风格约束

回复默认保持 **2-3 句以内的简短风格**，除非用户明确要求详细展开。同时适用于条形框气泡和面板对话。

# 5. MVP 范围与验收标准

## 5.1 MVP 使命

验证一个核心命题：

> 用户是否愿意使用一个桌面常驻的 AI Companion，来完成聊天、文件处理、轻记录和学习/工作陪跑。

## 5.2 MVP 核心能力

| 能力类别 | 具体内容 |
|---------|---------|
| **桌面入口** | 悬浮球常驻、可拖拽、三层交互（单击/双击/右键） |
| **对话能力** | 自然对话、承接上下文、流式输出、简短友好风格 |
| **文件处理** | 拖拽文件到悬浮球/条形输入框/面板 → 摘要 → 重点提炼 → 下一步建议（仅 .txt / .md / 代码文件）。所有入口共享同一对话与处理流程 |
| **轻陪跑** | 通过自然语言完成记录与轻待办（不提供结构化 UI），回顾近期内容，形成连续感 |

## 5.3 MVP 不做的事

- 复杂主动互动（不主动弹窗打扰）
- 多平台（仅 macOS）
- 高风险系统控制（不执行 shell 命令）
- 完整桌宠外观 / 动作系统
- 深度人格模拟 / 复杂关系机制
- 非文本文件（图片、PDF 等）

## 5.4 成功信号

如果 MVP 成立，用户应该会说出类似的话：
- "我会习惯性点它一下"
- "它在桌面上挺自然的"
- "它有点像我的小助手"
- "我愿意一直开着它"

具体验收标准待内测阶段根据实际情况制定。

# 6. 记忆与连续感设计

记忆是 Desktop-Claw 区别于一次性对话工具的核心能力。

## 6.1 记忆模型

采用 **Day-bucket（按天归档）** 模型：以自然日为单位，每天自动生成一份当日记忆摘要。

**一天的记忆包含：**

| 内容 | 说明 |
|------|------|
| **对话摘要** | 当天所有对话的关键要点（由 LLM 每日压缩） |
| **待办状态** | 用户创建的轻待办及完成情况 |
| **文件操作** | 处理过的文件名 + 摘要 |
| **偏好增量** | 当天新发现的用户偏好（如"喜欢简短回答"） |
| **今日一句话总结** | LLM 生成的当天概括 |

### Soul Prompt 机制

偏好增量不只是存档——每当发现新的用户偏好，需同步更新到 Claw 的 **Soul Prompt**（持久化的系统提示词）中。Soul Prompt 是 Claw 人格与风格的运行时表达，包含：

- 基础人格设定（语气、风格、边界）
- 累积的用户偏好（回复长度、称呼方式、关注领域等）
- 关系上下文（用户是谁、常做什么、近期状态）

偏好更新流程：日记忆压缩时提取偏好 → 与现有 Soul Prompt 合并去重 → 写回持久化存储。

## 6.2 记忆保留策略

- 近期记忆（7 天内）：完整保留，直接作为上下文
- 中期记忆（8-90 天）：压缩为每日摘要
- 长期记忆（>90 天）：暂定滚动淘汰（待用户测试后确定）

## 6.3 用户触达方式

- **被动触达**：对话中 Claw 自然引用近期记忆（"你昨天提到想写周报，写完了吗？"）
- **主动查看**：通过面板的日历/时间线视图浏览历史记忆（Milestone B）
- **搜索**：在面板中搜索过去的对话和文件（Milestone B）

## 6.4 "记得你"的具体表现（MVP）

- 当天对话有上下文连续性（不丢失前文）
- 当天创建的待办能在后续对话中引用
- 单击气泡能引用当日状态（"还有两件事没做完"，Milestone B 接入 LLM 后启用）

# 7. 产品人格

## 7.1 人格基调

| 维度 | 定位 |
|------|------|
| 整体感觉 | 温和、轻巧、有礼貌 |
| 个性程度 | 稍有个性，但不过度表演 |
| 陪伴感 | 愿意陪着用户，但不过分黏人 |
| 实用感 | 有帮助，但不说教 |

## 7.2 人格边界

不应该显得：过度拟人 → 出戏 | 过度热情 → 打扰 | 过度卖萌 → 失去可信度 | 过度工具化 → 没有角色感

在"有感情的存在感"和"克制的实用感"之间找平衡。

---

# 8. 设计原则

1. **Companion-first**：所有设计先问"这是否让它更像一个值得长期共处的桌面伙伴"。
2. **轻执行优先**：优先高频、轻量、低风险任务，不追求一步到位做复杂自动化。
3. **常驻但不打扰**：容易被长期打开，主动性和提醒应当节制。
4. **有用但不只有工具感**：必须有真实价值，但整体体验不能退化成"又一个效率工具"。
5. **先成立再成长**：先让用户愿意打开、愿意复用、愿意留下，再谈角色感和成长系统。

# 9. 竞品参考

**核心差异化**：Desktop-Claw 处于"桌面常驻 + AI 能力"的交叉点上，同时追求**陪伴感**和**轻实用性**。

- **ChatGPT Desktop / Raycast AI**：短路径效率工具，但无常驻感和陪伴层
- **传统桌宠**：有常驻感，但无 AI-Native 的理解能力
- **嵌入式 AI（Notion AI 等）**：依赖宿主应用，我们是独立桌面入口

# 10. 路线演进

## 阶段一：悬浮球小 Claw（当前）

**目标**：让最小产品体验成立——用户愿意开着它、愿意用它、不主动关掉它。

**产出物**：
- 悬浮球三层交互（单击/双击/右键）
- 基础对话（流式）+ 文件处理 + 轻记录
- 当日上下文连续性

**进入下一阶段的条件**：MVP 验收标准达标（§5.4）。

## 阶段二：Companion 感增强 + 陪伴探索

**目标**：让用户感知到"这是同一个伙伴"，并探索什么具体手段能真正提升陪伴感。

**产出物**：
- 跨天记忆（day-bucket 归档 + 按天回顾）
- Soul Prompt 持久化 + 稳定语气
- 日历/时间线视图
- 断线重连 + 错误恢复
- 陪伴感增强实验（具体方向待阶段二初期探索确定）

**进入下一阶段的条件**：用户能说出"它记得我"，且找到至少一个被验证有效的陪伴感增强手段。

## 阶段三：桌宠化演进

**目标**：基于阶段二的探索结论，从"有用的伙伴"进化为"有存在感的桌面角色"。

**可能方向**（取决于阶段二探索结果）：
- 角色外观 + 动作反馈
- 更丰富的互动方式
- 成长与关系感系统
- 更强的主动性（节制的主动提醒）

**进入条件**：阶段二完成 + 明确的用户需求信号。

# 11. 风险与待定事项

## 11.1 已知风险

| # | 风险 | 影响 | 当前应对 |
|---|------|------|---------|
| R1 | 单人开发带宽，Electron 全栈学习曲线 | 高 | 依赖 AI 辅助编码 |
| R2 | LLM API 费用不可控 | 中 | 用户自填 API Key，后续加用量提示 |
| R3 | Electron 打包体积 >100MB | 中 | 接受现状，MVP 后评估 Tauri |
| R4 | 桌面 AI Companion 用户接受度未知 | 高 | 内测 + DEVLOG 公开收集反馈 |
| R5 | macOS 沙盒限制文件读写 | 低 | 限定用户主动选择的目录 |

## 11.2 待定事项

| # | 问题 | 暂定方案 | 何时决策 |
|---|------|---------|---------|
| Q1 | Claw 语气风格偏可爱 vs 偏克制？ | 温和 + 不过分卖萌 | Milestone A 测试后 |
| Q2 | 记忆保留 7天/30天/永久？ | 90 天滚动窗口 | Milestone B 测试后 |
| Q3 | MVP 支持图片/PDF？ | 仅 .txt / .md / 代码 | Milestone A 前确定 |
| Q4 | 接入 Ollama / 本地模型？ | 仅 OpenAI-compatible | Milestone B 评估 |
| Q5 | Windows 支持时间线？ | 暂不排期 | 视社区需求 |
